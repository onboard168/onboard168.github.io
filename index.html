使用命令提示符 (CMD) 工具进行安装。

在 Windows 上将 Docker Desktop 安装到 D 盘并解决权限问题，其实是一个”曲线救国”的过程。因为 Docker 的官方安装包默认强制安装在 C:\Program Files\Docker\Docker，且没有提供直接的选择路径界面。

我们可以通过目录软链接（Symbolic Link）的方式，骗过安装程序，让它以为装在 C 盘，实则文件存储在 D 盘。

简要概述页面如何接收相机的流数据

发表于 2026-03-18 更新于 2026-04-18

使用到的技术栈

  • WebRtcStreamer：WebRTC 流媒体网关，用于协议转换
  • MediaMTX：开源媒体服务器，用于流媒体分发

数据流的产生流程

  1. 摄像头产生 RTSP 数据流
  2. MediaMTX 接收 RTSP 数据流
  3. WebRTC-Streamer 接收 MediaMTX 的数据流

架构概述

这套架构是目前工业级视频监控网页化的标准方案：

完整的数据流转路径

1. 产生源（Producer）

摄像头通过物理硬件捕获画面，编码成 H.264 原始数据，并运行一个 RTSP Server 进程。

2. 流媒体服务器（MediaMTX / Proxy）

MediaMTX 主动向摄像头拉取（Pull）RTSP 流，或者接收摄像头的推送（Push）。

作用：它像一个视频监控的”云盘”或”分发站”。它把摄像头的流稳定下来，支持多人同时拉取，避免多人直接访问摄像头导致摄像头 CPU 崩溃。

3. 协议转换层（WebRTC-Streamer）

WebRTC-Streamer 并不直接存储视频，它是一个 Gateway（网关）。

它从 MediaMTX 拉取 RTSP 流，实时地将 RTSP/RTP 包解封装，并重新打包成符合 WebRTC 标准的加密数据包（SRTP）。

4. 展示端（Browser）

浏览器通过 JS 调用 WebRtcStreamer.js 与后端握手，最终通过 UDP 直接接收来自 WebRTC-Streamer 的视频流。

为什么要加一层 MediaMTX？

你可能会问：”我直接用 WebRTC-Streamer 接摄像头不行吗？”当然可以，但加上 MediaMTX 有以下几个好处：

维度 直连 (Camera -> WebRTC-Streamer) 代理 (Camera -> MediaMTX -> WebRTC-Streamer)
并发能力 摄像头通常只允许 2-3 路连接，多了会掉线 MediaMTX 可以支持成百上千人同时看
协议转换 只能转 WebRTC 可以同时输出 RTSP, RTMP, HLS, WebRTC
稳定性 摄像头断网后前端连接直接死掉 MediaMTX 可以保持服务在线，等待摄像头重连
安全性 需要把摄像头的账号密码暴露在前端配置中 前端只访问 MediaMTX 的地址，隐藏了原始摄像头信息

有关VLN与VLA

发表于 2026-03-12

VLN 与 VLA

VLN（视觉-语言-导航）和 VLA（视觉-语言-动作）都是具身智能领域的核心技术方向，你可以把它们理解为一个以“移动”为目标，另一个以“操作”为核心。它们共享相同的“大脑”（视觉语言理解能力），但执行着不同的“身体”任务。

为了让你更清晰地理解，我把它们的核心区别和联系整理成下表：

对比维度 VLN (视觉-语言-导航) VLA (视觉-语言-动作)
🤔 核心定义 智能体结合视觉信息和自然语言指令，在环境中进行导航和移动的任务。 智能体结合视觉、语言信息，直接生成并执行具体的物理动作（如抓取、放置）的模型。
🎯 主要目标 理解指令并规划路径，从A点移动到B点（目标位置）。 理解指令并规划动作序列，完成对物体的精细操作。
🌰 典型任务 “去厨房，把放在餐桌上的红色杯子拿过来”（侧重于找到杯子）。 “拿起那个红色的杯子”（侧重于抓取杯子这个动作本身）。
⚙️ 输出形式 移动轨迹、线速度、角速度等导航指令。 机械臂关节角度、夹爪开合度等操作指令。
🗺️ 应用场景 扫地机器人、送货机器人、复杂环境下的自主探索。 家庭服务机器人（整理物品）、工业机械臂（精密装配）。

🤝 核心联系：VLM是它们共同的“大脑”

尽管VLN和VLA侧重点不同，但它们并非孤立存在，而是紧密相连的：

  1. 共享相同的“感官”：两者都基于视觉（Vision）和语言（Language）这两种核心模态，都需要理解“看到了什么”以及“指令说了什么”。
  2. VLM是共同的技术基石：视觉语言模型（VLM） 是这一切的基础。它就像机器人的“大脑”，负责建立视觉和语言之间的语义桥梁。
    • 在VLN任务中，VLM帮助智能体理解指令中“在沙发旁边”指的是哪个具体位置。
    • 在VLA任务中，VLM帮助机械臂理解“红色的杯子”是哪个目标物体。
  3. 共同构成完整的智能体：一个完整的具身智能机器人通常需要同时具备这两种能力。例如，当你命令机器人“去客厅拿一瓶水给我”时，它首先需要VLN能力导航到客厅，然后需要VLA能力识别并抓取水壶，最后可能还需要再次用VLN能力返回。从这个角度看，VLN可以看作是VLA在移动操作任务中的一个重要子集或前序步骤。

💡 一个比喻帮你理解

想象你有一个智能的AI管家，这三个概念的区别就非常清晰了：

  • VLM 就像是管家的认知和理解能力。你给他看一张聚会的照片，问他“大家都在干什么？”，他能回答你：“大家在客厅聊天、吃水果。” 他看懂了画面，并用语言描述出来。
  • VLN 就像是管家的认路和移动能力。你告诉他：“去二楼的卧室，把床头柜上的眼镜拿下来。” 他需要理解你的指令，然后自己规划路线，穿过走廊，爬上楼梯，准确找到卧室和床头柜的位置。
  • VLA 就像是管家的动手操作能力。当他找到了眼镜，就需要用这项能力，准确地伸出手，避开障碍物，轻柔地捏起眼镜，然后稳稳地拿给你。

Docker

发表于 2026-02-08 更新于 2026-03-05 分类于 devops

写在开头

在了解Docker 之前，我们先来聊聊为什么在已经有了虚拟机技术的情况下，我们还需要再造一个 Docker 容器技术?

首先，我们来举个生活场景的例子。如果把运行一个程序比作“我们要去住宿”，虚拟机就是给你盖了一栋独立别墅，而 Docker 则是给你提供了一间精装修的白领公寓。

对于每一栋别墅，也就是虚拟机（VM）都必须自己打地基、铺设独立的上下水管道、安装独立的电表（Guest OS操作系统、内核、虚拟硬件）；而docker，也就是所有的公寓共享大楼的整套地基、承重梁、主水管和供电系统（宿主机内核、CPU、内存）。那么作为入住的你，是愿意等待耗时更久的别墅，还是耗时较短的公寓？当然，这是仁者见仁，智者见智的事情，也许你愿意等待，毕竟独立的别墅，比较私密，不存在共享资源的问题，相对会更安全；但是如果你只是想要快速拎包入住，对安全没有那么高的要求，那么公寓也是一个不错的选择。

所以，如果程序运行的操作系统没有什么差别时，或者说所有的程序都基于统一操作系统运行，那么采用docker进行部署，就会比虚拟机更节省资源，也更高效。

Docker 镜像

前面提到docker的高效性，那么它为什么会更高效呢？这里就不得不提到docker镜像。

镜像（Image）是Docker运行容器的基础，镜像可以理解为Docker容器的模板，镜像可以基于其他镜像创建，也可以作为其他镜像的基础。

如果说容器是“运行中的公寓”，那么镜像（Image）就是那套极其高效的“装修图纸”和“建材包”。

Docker 之所以高效，绝不仅仅是因为它省去了虚拟机内核的开销，更核心的秘密在于它对镜像存储方式的革命性设计。它采用了联合文件系统 (UnionFS)，内部的核心是分层存储(Layering)模式。

这是 Docker 镜像最精妙的设计。想象一下你在装修公寓：

第一层（基础层）： 铺好地板（安装 Ubuntu 系统）。

第二层（环境层）： 搬进灶台（安装 JAVA 环境）。

第三层（应用层）： 摆放厨具（放入你的代码）。

这种分层带来的好处是惊人的：

复用与共享： 如果你启动了 10 个不同的 JAVA 应用，它们可以共用同一个 Ubuntu 镜像层和 JAVA 环境层（版本一样的情况下）。你的硬盘里只存了一份地板和灶台，而不是 10 份。

写时复制 (Copy-on-Write)： 所有的镜像层都是只读的。当你修改容器里的文件时，Docker 只会把需要修改的文件“复制”到最上面的容器运行层。这保证了底层镜像永远保持纯净，且修改非常轻量。

极速分发： 当你更新应用时，你只需要推送（Push）发生变化的那一层（代码层），底层庞大的系统镜像不需要重复传输。

Docker 容器

有了“装修图纸”和“建材包”，我们就可以快速构建公寓了，也就是可以基于镜像去创建容器。

镜像 (Image)是静态的、只读的。它是你打包好的“环境模板”。

容器 (Container)，是动态的、可读写的。它是在镜像这个“只读底座”上，加了一层可写层。
